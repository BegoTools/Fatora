# خطة تقوية Easy Store ERP ومضاهاة Easy Store Gold/Platinum (Pioneers)

> رفع المشروع لمستوى منتج **Easy Store Gold/Platinum** التجاري مع الالتزام الصارم بقيد المستخدم: **لا تكلفة علوية، لا سيرفر، كل البيانات لوكل على جهاز المستخدم.**
> مرجع المقارنة: pioneers-solutions.com/programs-easy-store-accounting-program (+ platinum).

---

## 0) القرارات الجذرية المُقفلة (Locked)

- **L1 — التوصيل:** تطبيق ويندوز عبر **Tauri** (يغلّف كود React الحالي → `.exe`/`.msi` ~10MB، مجاني/لوكل). يطابق نموذج البيع «دفعة واحدة + تثبيت». تكلفة بناء فقط: تثبيت Rust toolchain مرة على جهاز المطوّر؛ العميل يحصل على exe جاهز. يفتح وصول العتاد (باركود/طابعة حرارية).
- **L2 — التخزين:** **ملف JSON حقيقي على القرص** عبر Tauri fs (مثل `%APPDATA%/EasyStore/data.json`)، **لا** IndexedDB في الإنتاج. أسباب: IndexedDB داخل webview يمكن أن يُمحى بفعل «clear storage»/تحديث الـwebview → كارثة لبيانات محاسبية. الكتابة ذرية (atomic: write-temp → rename) + نسخ احتياطي. إعادة كتابة محصورة في طبقة `db` فقط (الـ AppState يبقى blob واحدًا).
- **L3 — الفروع:** **جهاز واحد مستقل** (متجر/فرع واحد لكل تثبيت) + **دمج دوري بالملف للقراءة فقط** (read-only rollup). المركز يستورد 1..N ملفات فروع → صفحة «تقارير مدمجة» تجمع الأرقام فقط، **بدون** دمج في البيانات العاملة (يتفادى تعارضات الدمج المدمّرة نهائيًا). لا مزامنة لحظية (تتطلب سيرفر = خارج النطاق).

## 1) أخطاء حرجة وُجدت في الكود الحالي (تُصلح أولًا)
- **مفتاح Gemini مكتوب في الكود** (`app/src/services/ai/gemini.ts` — `getFallbackKey`): يجب إزالته قبل أي توزيع. AI يصبح اختياريًا (المنتج يعمل بدونه عبر `localParser`).
- **كتابة fire-and-forget** (`db/index.ts:107` `saveState` و`AppContext.tsx` تستدعيها عند كل dispatch): خطأ/إغلاق بين dispatch والكتابة = فقد آخر العمليات. تُستبدل بكتابة ذرية متزامنة المفعول عبر Tauri fs.
- **`showToast` يستخدم `alert()`** (`AppContext.tsx:566`): سيئ UX. يُستبدل بـ`sonner` (مثبّتة وغير مستخدمة).
- **`LocalStorageRepository` غير مستخدمة** (`repository/index.ts`): تتضارب مع مسار IndexedDB الحقيقي. تُحذف أو تُعاد توجيهها للملف الجديد.
- **عدم وجود `updatedAt`/`deletedAt` عام** يحدّد دمج الفروع لاحقًا — يُضاف على الأقل للكيانات الرئيسية في v2.

## 2) النطاق: v1 قابل للبيع مقابل v2 مقابل خارج النطاق

### v1 (MVP قابل للبيع — المراحل 0+1+2+6+التعبئة)
منتج ويندوز آمن + ترخيص + فواتير ضريبية + Excel + نسخ احتياطي. هذا أول مخرج تجاري.

### v2 (دفع المضاهاة الكاملة — المراحل 3+4+5+7+دمج الفروع)
مخازن متعددة، حسابات متقدمة، مبيعات/مشتريات متقدمة، تقارير مالية، دمج فروع.

### خارج النطاق صراحةً
- أي سيرفر/DBMS خارجي/استضافة سحابية (قيد المستخدم).
- ربط ETA المصري / ZATCA السعودي الفعلي (يحتاج خادم + شهادة توقيع + تكلفة) — نفذ فواتير **متوافقة الشكل** فقط.
- مزامنة فروع لحظية عبر الإنترنت (تتطلب سيرفر).
- تطبيق موبايل أصلي / بوابة دفع اشتراكات SaaS.

## 4) تدفق البيانات (Data Flow) بعد v1
```
UI (React) → dispatch(reducer) → AppState(ذاكرة) → saveState() ──atomic write──▶ data.json
                                                  → backup rotation (يومي + يدوي)
                                                  → auditLogs (سجل تدقيق، موجود)
ترخيص: license.ts (بصمة+HMAC) → تقييد الكتابة عند expiry (قراءة فقط)
دمج فروع (v2): ملفات .esbranch ──load──▶ ذاكرة عرض فقط ──▶ تقارير مدمجة
```

## 5) المخاطر والتخفيف
| الخطر | التخفيف |
|-------|---------|
| فقدان بيانات محاسبية بمسح تخزين webview | L2: ملف JSON على القرص + كتابة ذرية + نسخ احتياطي |
| فقد آخر العمليات بـfire-and-forget | كتابة ذرية متزامنة المفعول (1.2/0.1) |
| ربط ETA/ZATCA يكسر القيد | خارج النطاق؛ فواتير متوافقة الشكل فقط |
| ربط فروع لحظي يحتاج سيرفر | L3: دمج دوري للقراءة فقط بالملف |
| ترخيص محلي يُكسر بسهولة | كافٍ للحد من النسخ السهل؛ ليس DRM كامل (مقابل تعقيد/تكلفة) |
| ترحيل بيانات المستخدم الحالي من IndexedDB | ترحيل تلقائي عند أول إقلاع تحت Tauri (1.3) |
| حجم بيانات كبير لاحقًا | الملف يكفي لمعظم المتاجر الصغيرة/المتوسطة؛ SQLite علائقية مؤجلة كخيار v3 |

## 6) التحقق (Validation)
- بعد كل مرحلة: `npm run lint && npm run typecheck && npm run build` نظيفة + Vitest خضراء.
- سيناريوهات محاسبية: بيع بضريبة شاملة + مرتجع جزئي → رصيد العميل/الخزينة/VAT المستحقة صحيحة؛ نسخ احتياطي + استعادة سليمة؛ تفعيل ترخيص صحيح/خاطئ/منتهي على جهازين (بصمة مختلفة)؛ تثبيت `.exe` على ويندوز نظيف + عمل offline كامل؛ ترحيل بيانات قديمة من IndexedDB دون فقد؛ استيراد Excel بقالب خاطئ → rollback دون تلويث البيانات.
- دمج فروع (v2): ملفا فرعين بنفس معرّف عميل → صفحة التقارير المدمجة تجمع الرصيد دون تعديل بيانات الفرع المحلي.

## 7) نقاط تفوق مقترحة على المرجع (ميزات بيع)
- مساعد AI متكامل (Gemini + fallback محلي) — Easy Store لا يملكه.
- عربي/إنجليزي + RTL أصيل + تصميم Azimuth حديث.
- Offline-First حقيقي + تثبيت أخف (Tauri ~10MB).
- أمان بيانات على مستوى القرص (ملف ذري + نسخ احتياطي) بدل تخزين متصفح هش.

## 8) أسئلة مفتوحة (لا تعيق البدء بـ v1)
- هل تملك/تريد شهادة توقيع ويندوز للـexe؟ (ثقة تثبيت فقط، غير مانع).
- السوق مصر فقط (VAT 14%) أم مصر+سعودية (ZATCA)؟ يحدد صيغة QR الافتراضية (TLV مصري الآن). إن سعودي — تنفيذ ZATCA Phase-2 QR (لا يزال بدون ربط حكومي، يكفي للشكل المعتمد) في v1.
- هل يُتوقع حجم بيانات ضخم جدًا (مئات آلاف الأصناف)؟ إن نعم — تقديم SQLite علائقية مبكرًا في v1 بدل الملف.
